# Deployment — BagelBoyzNJ.com

## Hosting
- **Host:** Hostinger shared hosting
- **Domain:** bagelboyznj.com
- **Server webroot:** `public_html/`
- **Repo:** https://github.com/somarmedia/BagelBoyzNJ (branch `main`)

## Normal workflow — Git auto-deploy

Hostinger is configured (in hPanel → Advanced → Git) to pull from `main` and deploy into `public_html/`.

**Shipping a change is just:**

```bash
git add <files>
git commit -m "..."
git push origin main
```

Within ~seconds of the push, the change is live at bagelboyznj.com. No FTP needed.

**Verify a deploy:** `curl https://bagelboyznj.com/<changed-file>` or just reload the page. For PHP endpoints, `curl -X POST` the endpoint and check the response.

## Files that do NOT auto-deploy

Anything in `.gitignore` is not pushed to GitHub and therefore not picked up by auto-deploy. These need **manual upload** via Hostinger File Manager (hPanel → Files → File Manager) or FTP:

| Local path                          | Server path                           | Purpose |
|-------------------------------------|---------------------------------------|---------|
| `php/smtp-config.php`               | `public_html/php/smtp-config.php`     | SMTP credentials for form handlers |
| `backend/.env`                      | (not on website server — backend runs elsewhere) | Backend API keys |
| `social-feed.json`                  | `public_html/social-feed.json`        | Auto-generated by social autopilot; pushed by backend, not dev |

**Never commit these to git** — `.gitignore` protects them. If you need a teammate to recreate one, share the password/keys out-of-band (not in chat, not in the repo).

## First-time setup for a new secret file

1. Create it locally at the path listed above
2. Confirm it's gitignored: `git check-ignore <path>` should print the path and exit 0
3. Upload via Hostinger File Manager to the matching server path
4. Test the feature that uses it

## Site structure on the server

```
public_html/
├── index.php, menu.php, catering.php, etc.  ← auto-deployed from repo root
├── css/, js/, img/, includes/               ← auto-deployed
├── php/
│   ├── catering-inquiry.php                 ← auto-deployed
│   ├── careers-apply.php                    ← auto-deployed
│   ├── PHPMailer/                           ← auto-deployed (committed to repo)
│   └── smtp-config.php                      ← MANUAL upload only (gitignored)
└── social-feed.json                         ← pushed by backend worker
```

## Form handler SMTP config

The form handlers (`php/catering-inquiry.php`, `php/careers-apply.php`) authenticate to Hostinger SMTP using credentials in `php/smtp-config.php`. Structure is in `php/smtp-config.example.php` (committed). See [php/smtp-config.example.php](php/smtp-config.example.php) for the template.

If `smtp-config.php` is missing on the server, both handlers return a graceful JSON error instead of a 500. Debug by hitting the endpoint directly:

```bash
curl -X POST https://bagelboyznj.com/php/careers-apply.php \
  -d "name=Test&phone=5555555555&age=18&position=Counter / Cashier&preferred_location=Holmdel Rd&availability=Full-Time"
```

## Debugging a broken deploy

1. **Did it push to GitHub?** `git log origin/main -1` should match local `git log -1`
2. **Did auto-deploy fire?** hPanel → Advanced → Git → check deployment log
3. **Is the file on the server?** hPanel → File Manager → navigate to `public_html/...`
4. **PHP error?** hPanel → Advanced → PHP Error Log, or look at `public_html/error_log` via File Manager
5. **Browser-side error?** DevTools → Network tab → click failing request → Response tab

## Backend (social automation)

The Node.js backend in `/backend/` does **not** run on Hostinger shared hosting. It runs separately (PM2 on a VPS or similar) and pushes `social-feed.json` to the public webroot. Backend deploy is a separate concern from the website deploy above.
